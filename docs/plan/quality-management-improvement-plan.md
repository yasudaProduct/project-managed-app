# 定量品質管理機能 改善計画

作成日: 2026-04-19
対象ブランチ: `claude/add-quality-metrics-BJqhO`

## 背景

現状の定量品質管理機能は「品質メトリクスの収集・可視化基盤」として最低限のレベルが整っているが、ソフトウェア定量品質"管理"として現場運用するには不足機能がある。本ドキュメントは、レビュー結果をもとに段階的な機能拡張方針をまとめる。

## 現状の機能カバレッジ

| # | 観点 | 状態 |
|---|------|------|
| 1 | 対象成果物の分類 (QualityDocumentType / QualityReviewType) | 実装 |
| 2 | 工程(フェーズ)別管理 | 部分的(集計軸として提供、閾値は工程別ではない) |
| 3 | レビュー指標 (レビュー密度) | 実装 |
| 4 | テスト指標 (指摘密度 / Major指摘密度) | 実装 |
| 5 | 欠陥分類 (ODC) | 未実装 (category はフリーテキスト) |
| 6 | 閾値の運用 | 部分的(プロジェクト一律、工程別/成果物別なし) |
| 7 | 判定結果の活用 (NORMAL/WARNING/DANGER) | 部分的(表示のみ、是正アクションなし) |
| 8 | 傾向分析 (日次推移、累積) | 部分的(信頼度成長曲線・パレート分析なし) |
| 9 | 帳票・報告 (TSV/CSV エクスポート) | 実装 (PDF 品質報告書なし) |
| 10 | PDCA サイクル | 未実装 |

## 改善計画 (優先度順)

### 優先度A — 品質"管理"として機能するために必須

#### A-1. 欠陥分類体系 (ODC: Orthogonal Defect Classification) の整備

**現状:** `QualityFinding.category` はフリーテキストのみで、集計・比較が不可能。

**改善内容:**
- 混入工程 enum (REQUIREMENT / DESIGN / CODING / TEST 等) フィールドを追加
- 検出工程 enum (DESIGN_REVIEW / CODE_REVIEW / UNIT_TEST / INTEGRATION_TEST / SYSTEM_TEST 等) フィールドを追加
- 欠陥種別 enum (SPEC_MISSING / DESIGN_ERROR / CODING_ERROR / TEST_MISSING / DOCUMENT / OTHER) フィールドを追加
- これにより「どの工程で混入した欠陥が、どの工程で検出されたか」の根本原因分析が可能になる

**影響範囲:**
- `prisma/schema.prisma` (QualityFinding モデルに列追加 + enum)
- `src/domains/quality/value-objects/quality-enums.ts`
- `src/domains/quality/quality-finding.ts`
- CSV インポート / TSV エクスポート (列追加)
- UI (指摘登録フォーム / 表示)

#### A-2. 是正アクション管理

**現状:** 判定 (DANGER / WARNING) 後のアクションがシステムに記録できない。計測値は見られるが、PDCA の「CA (Check / Act)」が機能しない。

**改善内容:**
- `QualityCorrectiveAction` エンティティ追加
  - 対象: WbsId / TargetId / FindingId / 指標の閾値超過
  - 項目: 原因分析 / 対応施策 / 担当者 / 期限 / ステータス (OPEN / IN_PROGRESS / DONE / VERIFIED)
  - 効果測定: 再評価時の指標値を記録
- ダッシュボードから「このDANGER指標に是正アクションを登録」導線

**影響範囲:**
- Prisma スキーマ新規テーブル
- Domain / Application / Infrastructure / UI 新規一式

#### A-3. レビュー指標の補完

**現状:** レビュー密度 (レビュー工数 / 規模) のみ。一般的な指摘密度・改修率が分離していない。

**改善内容:**
- レビュー指摘密度 (件 / page) を独立指標として追加 (現状の defectDensity と混同しやすい)
- 指摘改修率 (指摘に対する対応済み件数の割合)
- 再レビュー率 (指摘件数が多く再レビューを要した対象の比率)

**影響範囲:**
- `QualityFinding` に対応状態 (status: OPEN / FIXED / WONTFIX 等) を追加
- `QualityMetricsCalculator` 新メソッド
- ダッシュボード指標カード追加

---

### 優先度B — 管理精度を高めるために追加したい

#### B-1. 工程別閾値

**現状:** 閾値はプロジェクトに1セット。設計レビューと単体テストは期待値が全く異なるため、適切な判定ができない。

**改善内容:**
- `QualityThreshold` を {projectId, phaseId} 単位で保持可能に
- 閾値管理 UI でフェーズごとに設定
- ダッシュボードの判定はフェーズ別閾値を自動参照

#### B-2. パレート分析

**現状:** 指摘の原因/種別ごとの累積寄与度が可視化できない。

**改善内容:**
- A-1 の欠陥分類実装後、カテゴリ別の件数降順グラフ + 累積%ライン (80/20 可視化)
- 「どのカテゴリを潰せば全体の8割を改善できるか」を表示

#### B-3. 成果物タイプ別の指標管理

**現状:** `QualityDocumentType` は定義済みだが、ダッシュボードで成果物別に切り替えて見られない。

**改善内容:**
- ダッシュボードに「成果物タイプ」フィルタを追加
- 「設計書のレビュー密度」「コードのレビュー密度」を分離表示

---

### 優先度C — 追加価値として欲しい

#### C-1. 品質報告書の PDF 出力

**現状:** TSV/CSV エクスポートのみで、ステークホルダー向け資料としては使えない。

**改善内容:**
- 指標サマリ + 判定結果 + 是正アクション状況を含む品質報告書 PDF 出力
- テンプレート設計 (エグゼクティブサマリ / 詳細セクション)

#### C-2. 信頼度成長曲線

**現状:** 日次推移は実績のみ。予測モデルなし。

**改善内容:**
- Rayleigh / Gompertz モデルで指摘検出の収束予測
- 「あとXX件程度で検出がほぼ完了する見込み」という可視化

#### C-3. プロジェクト横断比較

**現状:** 当該WBSの数値のみ。自社ベースラインとの対比不可。

**改善内容:**
- 組織の過去プロジェクト平均値を算出
- 「今回の指摘密度は社内平均の N倍」のような相対評価

---

## 優先度外 / 技術的改善 (別タスク)

レビュー時に検出した一般的改善点。機能追加ではなく内部品質向上のため、PDCA 強化とは切り離して優先度を判断する。

- セキュリティ: `quality-actions.ts` の Server Action に WBS/プロジェクトメンバ認可チェックがない
- パフォーマンス: `getTaskManHours` の N+1、 `registerSizeMetric` の逐次処理
- データ整合性: `QualityReviewTarget` と WbsTask の論理結合キーに FK 制約がない
- 閾値検証: `QualityThreshold.validate()` で `warn > danger` の矛盾検出が必要
- UI 状態: sizeUnit が `useState` でリロード時に初期化。URL パラメータ化が望ましい

## 着手順序の推奨

1. **A-1 (欠陥分類体系)** — 他改善の基盤。これがないと A-2 / B-2 が意味を持たない
2. **A-3 (レビュー指標補完)** — スキーマ拡張が小さく効果が大きい
3. **A-2 (是正アクション管理)** — PDCA の要。スキーマ新規追加が大きい
4. **B-1 (工程別閾値)** — 運用現場からの要望が高い
5. **B-2 (パレート分析)** — A-1 実装後にデータが揃い次第
6. **B-3 / C-1 〜 C-3** — 運用成熟度に応じて段階的に
