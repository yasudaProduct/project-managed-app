# 営業日按分ロジック仕様書

## 1. 概要

本仕様書は、WBSタスクの工数按分に用いる**月別稼働可能時間の比率計算ロジック**を定義する。
会社カレンダー（土日・祝日・会社独自休日）と個人スケジュール（休暇・会議等の予定）を考慮し、担当者ごとに月別の稼働可能時間を算出する。この比率を [04: 月別工数按分](./04-monthly-allocation.md) で使用することで、タスク工数が各月へ配分される。

> **関連仕様書との位置づけ:**
> - 本仕様書（01）: **「いつ・何時間稼働できるか」を計算する**（カレンダードメイン）
> - [04: 月別工数按分](./04-monthly-allocation.md): **「その比率をタスク工数に適用・集計する」**（WBSドメイン）

対象となる工数は以下の3種類であり、それぞれ独立して月別按分が行われる。

| 工数種別   | フィールド名   | 説明                           |
| ---------- | -------------- | ------------------------------ |
| 予定工数   | `yoteiKosu`    | 当初計画の工数                 |
| 基準工数   | `kijunKosu`    | スコープ変更前の基準となる工数 |
| 見通し工数 | `forecastKosu` | 最新の完了見通し工数           |

## 2. 用語定義

### 稼働可能時間 (Available Hours)

ある1日において担当者がタスク作業に充てられる時間数。次の優先順位で計算される。

```
稼働可能時間 = min(個人予定控除後の時間, 稼働率上限時間)
個人予定控除後の時間 = max(0, 基準時間 - 個人予定合計時間)
稼働率上限時間 = 基準時間 × 参画率
```

### 基準時間 (Standard Working Hours)

会社カレンダーに設定された1日あたりの標準稼働時間数（例: 8時間、7.5時間）。
`CompanyCalendar` のコンストラクタ引数 `standardWorkingHours` で指定する。

### 営業日 (Business Day)

担当者の稼働可能時間が1時間以上ある日。以下の条件をすべて満たす日が営業日となる。

1. 会社カレンダー上の休日（土日・祝日・会社独自休日）でない
2. 個人スケジュールに全日休暇が登録されていない
3. 担当者の参画率が0より大きい

### 全日休暇 (Full Day Off)

個人スケジュールのタイトルが特定キーワードと完全一致する予定。全日休暇が登録された日は非稼働日として扱われ、稼働可能時間は0になる。

### 参画率 / 稼働率 (Rate)

担当者がこのタスクに参加する割合（0.0〜1.0）。参画率が0.5の場合、1日あたりの稼働可能時間は基準時間の50%が上限となる。

### 月別按分比率 (Monthly Allocation Ratio)

ある月の稼働可能時間合計 ÷ 期間全体の稼働可能時間合計。この比率を工数総量に掛けることで各月の按分工数を算出する。

### 単月タスク (Single Month Task)

予定開始日と予定終了日が同一年月に収まるタスク、または予定終了日が未設定のタスク。按分計算は行わず、全工数を開始月に計上する。

### 基準工数期間 (Kijun Period)

基準工数 (`kijunKosu`) の按分に用いる期間。予定期間とは独立して指定される（`kijunStart` / `kijunEnd`）。

## 3. 稼働可能時間の計算ルール

稼働可能時間の計算は以下の優先順位で行う。上位の判定でfalseになった場合、後続の判定は行わない。

```
Step 1: 会社カレンダー休日判定
  → 土曜・日曜・祝日・会社独自休日の場合: 稼働可能時間 = 0（非稼働日）

Step 2: 個人スケジュール全日休暇判定
  → 全日休暇キーワードと一致するスケジュールがある場合: 稼働可能時間 = 0（非稼働日）

Step 3: 参画率チェック
  → 参画率 = 0 の場合: 稼働可能時間 = 0（非稼働日）

Step 4: 稼働可能時間の計算（稼働日の場合）
  scheduledHours = 個人予定の合計時間（上限 = 基準時間）
  rawAvailable   = max(0, 基準時間 - scheduledHours)
  rateCapHours   = 基準時間 × 参画率
  availableHours = min(rawAvailable, rateCapHours)
```

## 4. 会社休日の判定条件

`CompanyCalendar.isCompanyHoliday(date)` が `true` を返す条件を以下に示す。
3つの条件はOR判定であり、いずれか1つに該当すれば休日となる。

| 優先順位 | 判定内容                                    | 実装                                |
| -------- | ------------------------------------------- | ----------------------------------- |
| 1        | 土曜日（`date.getDay() === 6`）             | 組み込み                            |
| 2        | 日曜日（`date.getDay() === 0`）             | 組み込み                            |
| 3        | 日本の祝日                                  | `@holiday-jp/holiday_jp` ライブラリ |
| 4        | 会社独自休日（`CompanyHoliday` リスト照合） | 年月日の文字列一致（YYYY-MM-DD）    |

### 会社独自休日 (CompanyHoliday)

会社独自休日は日付・名称・種別（`NATIONAL` / `COMPANY` / `SPECIAL`）の3フィールドを持つ。
日付照合は `YYYY-MM-DD` 形式の文字列比較で行うため、時刻情報は無視される。

### テスト例

```
2026-05-23（土） → 休日
2026-05-24（日） → 休日
2026-05-25（月） → 稼働日
2026-01-01（元日） → 休日（祝日ライブラリ）
2026-02-11（建国記念の日） → 休日（祝日ライブラリ）
2026-12-29（会社独自休日に登録済み） → 休日
2026-12-28（会社独自休日に未登録の平日） → 稼働日
```

## 5. 全日休暇の判定ルール

`AssigneeWorkingCalendar` 内の `isFullDayOff(schedule)` は、スケジュールのタイトルが以下のキーワードと**完全一致**する場合に全日休暇と判定する。

### 定義場所

キーワードは [`src/domains/calendar/assignee-working-calendar.ts`](../../src/domains/calendar/assignee-working-calendar.ts#L96) にハードコードされている。

```
['休暇', '有給', '休み', '全休', '代休', '振休', '有給休暇']
```

> 将来的にはプロジェクト設定から動的に取得する想定（実装上は `TODO` コメントあり）。

**完全一致**であるため、「午前休」「半休」「有給取得」などのタイトルは全日休暇とみなされない。これらは部分休暇として時間控除の対象になる（後述）。

また、判定は登録されたスケジュールの `userId` と担当者の `userId` が一致するものだけに適用される。別ユーザーの休暇スケジュールは影響しない。

## 6. 個人予定による控除計算

### 全日休暇の場合

全日休暇キーワードに完全一致するスケジュールが存在する日は、`isWorkingDay` が `false` を返すため、稼働可能時間は0となる。その日は按分の分母となる稼働可能時間に算入されない。

### 部分休暇（時間控除）の場合

全日休暇以外のスケジュール（会議・出張・研修など）は時間控除の対象となる。控除時間は `startTime` と `endTime`（"HH:mm" 形式）から計算される。

```
控除時間 = (endHour + endMin / 60) - (startHour + startMin / 60)
```

同日に複数の予定がある場合は合計を計算し、基準時間を上限にクランプする。

```
scheduledHours = min(基準時間, 全予定の控除時間合計)
例：予定の合計が 10時間 → min(7.5, 10) = 控除時間は7.5
```

全日休暇と時間控除予定が同日に混在する場合は、全日休暇が優先され `scheduledHours = 基準時間` となる（非稼働日扱い）。

### 計算例

基準時間 = 7.5時間、参画率 = 1.0 の場合:

| シナリオ               | 予定                     | 個人予定合計時間 (scheduledHours) | 個人予定控除後の時間 (rawAvailable) | 稼働率上限時間 (rateCapHours) | 稼働可能時間 (availableHours) |
| ---------------------- | ------------------------ | --------------------------------- | ----------------------------------- | ----------------------------- | ----------------------------- |
| 予定なし               | なし                     | 0                                 | 7.5                                 | 7.5                           | 7.5                           |
| 2時間会議              | 10:00-12:00              | 2                                 | 5.5                                 | 7.5                           | 5.5                           |
| 1時間+1.5時間          | 10:00-11:00, 14:00-15:30 | 2.5                               | 5.0                                 | 7.5                           | 5.0                           |
| 5時間出張（参画率0.8） | 09:00-14:00              | 5                                 | 2.5                                 | 6.0                           | 2.5                           |
| 終日出張+懇親会        | 09:00-17:00, 18:00-20:00 | 7.5（上限クランプ）               | 0                                   | 7.5                           | 0                             |
| 有給（全日）           | title="有給"             | —                                 | —                                   | —                             | 0（非稼働日）                 |
| 45分会議               | 09:30-10:15              | 0.75                              | 6.75                                | 7.5                           | 6.75                          |

## 7. 月別営業日按分アルゴリズム

### アルゴリズム概要

按分処理が実行する手順:

```
Step 1: 期間内の各日について稼働可能時間を計算し、月別に合算する
        → monthlyAvailableHours: Map<"YYYY/MM", number>

Step 2: 期間全体の稼働可能時間合計を算出する
        → totalAvailableHours = sum(monthlyAvailableHours.values())

Step 3: totalAvailableHours = 0 の場合はエッジケース処理（後述）

Step 4: 各月の按分比率を計算し、按分工数を算出する
        → ratio = monthAvailableHours / totalAvailableHours
        → allocatedHours = totalHours × ratio
        → 小数第2位で丸め（Math.round(x * 100) / 100）

Step 5: 丸め誤差を最終月に吸収する
        → difference = totalHours - sum(distributedHours)
        → 最終月の工数 += difference
        → 再度小数第2位で丸め
```

### 丸め処理

各月の按分工数は小数第2位で丸める。複数月の丸め誤差は最終月（月キーの辞書順で最後）で吸収されるため、全月の按分工数の合計は必ずもとの `totalHours` と一致する（小数第2位精度）。

### 月キーのフォーマット

月を表すキーは `"YYYY/MM"` 形式（例: `"2026/05"`）。

## 8. エッジケース一覧

### 稼働可能時間合計 = 0 の場合

期間全体を通じて稼働可能時間がゼロの場合（参画率0、または全日程が休日・休暇）、**開始月に全工数を計上**する。

```
// 期間: 2026-05-25〜2026-06-05、参画率0の担当者
// → result: { "2026/05": 100 }
```

### 単月タスク（予定期間が同一年月に収まる場合）

単月タスクと判定された場合、按分計算を行わず全工数を開始月に計上する。

```
// 期間: 2026-05-25〜2026-05-29（同一月）
// → allocatedPlannedHours: { "2026/05": 40 }
```

> 基準工数の按分においても同じルールが適用される。`kijunStart` と `kijunEnd` が同一年月に収まる場合、按分を行わずその月に全額計上する。

### 予定終了日が未設定の場合

`yoteiEnd` が `undefined` の場合、`yoteiStart` と同じ日を終了日とする（単日タスクとして扱う）。
また、`yoteiEnd` が未設定のタスクは単月タスクとして扱われる。

### 工数 = 0 の場合

`distributeHoursByBusinessDays(0)` を呼んだ場合、各月の按分工数はすべて0になる。

### 見通し工数 = 0 または未設定の場合

`forecastKosu` が `0` または `undefined` の場合は見通し工数の按分を実行しない。全月の `allocatedForecastHours` は0のまま。

### 基準期間が単月の場合

基準工数の按分において、`kijunStart` と `kijunEnd` が同一年月に収まる場合はその月に全額計上する（`BusinessDayPeriod` を使わない）。

### 担当者未割当の場合

`allocateTaskWithDetails` で `assignee` が `undefined` の場合、`WbsAssignee.createUnassigned(wbsId)` でダミー担当者を生成して按分を実行する。

## 9. 計算例

### 例1: 単月タスク（按分なし）

- 期間: 2026-05-25（月）〜 2026-05-29（金）
- 予定工数: 40時間
- 参画率: 1.0
- 基準時間: 8時間/日

```
5月の営業日: 月・火・水・木・金 = 5日
5月の稼働可能時間: 5日 × 8時間 = 40時間
totalAvailableHours = 40（単月）

按分結果:
  2026/05: 40時間（全額）
```

### 例2: 複数月タスク（等分案分）

- 期間: 2026-05-25（月）〜 2026-06-05（金）
- 予定工数: 100時間
- 参画率: 1.0
- 基準時間: 8時間/日

```
5月の営業日（期間内）: 25・26・27・28・29 = 5日 → 40時間
6月の営業日（期間内）: 1・2・3・4・5 = 5日 → 40時間
totalAvailableHours = 80時間

5月比率: 40 / 80 = 0.5 → 100 × 0.5 = 50時間
6月比率: 40 / 80 = 0.5 → 100 × 0.5 = 50時間

按分結果:
  2026/05: 50時間
  2026/06: 50時間
```

### 例3: 参画率0.5の担当者

- 期間: 2026-05-25（月）〜 2026-05-29（金）
- 予定工数: 40時間
- 参画率: 0.5
- 基準時間: 8時間/日

```
5月の稼働可能時間: 5日 × (8 × 0.5)時間 = 5日 × 4時間 = 20時間

按分結果（単月）:
  2026/05: 40時間（全額）
```

> 参画率は月別按分の比率（ratio）には影響しないが、稼働可能時間の絶対値には影響する。
> ただし複数月の場合に同じ参画率が均等に適用されれば比率は変わらず、結果の按分比率は参画率によって変化しない。
> 月によって稼働日数が異なる場合（祝日・休暇など）には比率に差が生じる。

### 例4: 個人予定（有給）による按分比率の変化

- 期間: 2026-05-25（月）〜 2026-06-05（金）
- 予定工数: 100時間
- 参画率: 1.0
- 基準時間: 8時間/日
- 個人予定: 2026-05-26（火）に有給（全日休暇）

```
5月の稼働可能時間:
  25(月)=8, 26(火)=0（有給）, 27(水)=8, 28(木)=8, 29(金)=8
  合計 = 32時間

6月の稼働可能時間:
  1(月)=8, 2(火)=8, 3(水)=8, 4(木)=8, 5(金)=8
  合計 = 40時間

totalAvailableHours = 72時間

5月比率: 32 / 72 ≈ 0.4444 → 100 × 0.4444 ≈ 44.44時間
6月比率: 40 / 72 ≈ 0.5556 → 100 × 0.5556 ≈ 55.56時間
（丸め誤差は最終月で吸収）

按分結果:
  2026/05: ~44.44時間（有給なし時の50時間より少ない）
  2026/06: ~55.56時間（有給なし時の50時間より多い）
```

### 例5: 丸め誤差の吸収

- 総工数: 123.45時間
- 3月にまたがる期間（各月の稼働可能時間が不均等と仮定）

```
各月の按分後に丸め誤差が生じた場合:
  全月の按分合計 ≠ 123.45 になることがある

→ 最終月（Map内の最後のキー）に difference を加算
→ 最終的な全月合計 = 123.45（小数第2位精度）
```

### 例6: 3ヶ月タスクの工数按分（2024年1月〜3月）

- 期間: 2024-01-01〜2024-03-31
- 予定工数: 100時間
- 見通し工数: 120時間
- 参画率: 1.0（基準時間: 7.5時間/日）

```
各月の稼働可能時間は実際の営業日数 × 7.5時間で決まる（祝日を除く）。

按分の仕組み:
  各月: ratio = 月の稼働可能時間 / 全月の稼働可能時間合計
  予定工数按分 = 100 × ratio
  見通し工数按分 = 120 × ratio

全月の予定工数按分合計 = 100時間（丸め誤差は最終月で吸収）
全月の見通し工数按分合計 = 120時間（丸め誤差は最終月で吸収）
```

## 10. 基準工数(KIJUN)と見通し工数(forecast)の同時按分

### 概要

タスク工数の月別按分は、1回の呼び出しで予定工数・見通し工数・基準工数の3種類を同時に処理する。

### 各工数の按分方法

| 工数種別                    | 按分期間                             | 条件                                                             |
| --------------------------- | ------------------------------------ | ---------------------------------------------------------------- |
| 予定工数 (`yoteiKosu`)      | 予定期間（`yoteiStart`〜`yoteiEnd`） | 常に按分                                                         |
| 見通し工数 (`forecastKosu`) | 予定期間（`yoteiStart`〜`yoteiEnd`） | `forecastKosu` が設定されており > 0 の場合                       |
| 基準工数 (`kijunKosu`)      | 基準期間（`kijunStart`〜`kijunEnd`） | `kijunStart`, `kijunEnd`, `kijunKosu` がすべて設定されている場合 |

### 見通し工数の按分

見通し工数は予定期間と同じ稼働可能時間比率で按分される（予定工数と同じ按分比率を基準にする）。

- `forecastKosu` が `undefined` または `0` の場合: 按分しない（全月で `allocatedForecastHours = 0`）
- 単月タスクの場合: 全額が開始月の `allocatedForecastHours` に計上される

```
// 単月タスクの場合
{
  month: "2024/01",
  allocatedPlannedHours: 50,   // yoteiKosu
  allocatedForecastHours: 60,  // forecastKosu
}
```

### 基準工数の按分

基準工数は基準期間（`kijunStart`〜`kijunEnd`）で独立して按分される。

- 基準期間が単月の場合: その月に全額計上
- 基準期間が複数月の場合: 基準期間の稼働可能時間比率で按分

予定工数の按分月と基準工数の按分月は一致しない場合がある（期間が異なるため）。

### 量子化器

オプションで量子化器を指定することで、按分後の工数を特定単位（例: 0.5時間単位）に丸めることができる。量子化器は**予定工数・見通し工数・基準工数の3種類すべて**に適用される。

量子化器が指定された場合、各工数種別ごとに1回呼び出される（複数月タスクで予定工数・見通し工数の両方が設定されている場合は2回呼び出される）。

### 担当者未割当時の動作

`assignee` が未設定の場合、参画率1.0のダミー担当者を生成して按分を実行する。工数合計の按分結果は参画率1.0の通常担当者と同一になる。
