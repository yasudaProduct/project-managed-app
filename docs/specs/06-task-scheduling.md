# タスクスケジューリング仕様書

## 1. 概要

本仕様書は、WBSに登録されたタスクの予定開始日・予定終了日を**前詰め方式**で自動計算するタスクスケジューリング機能を定義する。

担当者ごとの稼働カレンダー（会社カレンダー + 個人スケジュール）を考慮し、タスクの予定工数を消化できる期間を算出する。

2つのモードを提供する。

| モード | 用途 |
| --- | --- |
| **初期計画モード** | プロジェクト立ち上げ時に全タスクの日程を一括計算する |
| **リスケモード** | プロジェクト進行中に未完了タスクの日程を引き直す |

> **関連仕様書:**
> - [01: 営業日按分ロジック](./01-working-hours-allocation.md): 稼働可能時間の計算ロジック（CompanyCalendar / AssigneeWorkingCalendar）

---

## 2. 用語定義

| 用語 | 説明 |
| --- | --- |
| 前詰め | 起点日から担当者ごとにタスクを隙間なく割り当てるスケジューリング方式 |
| 稼働カレンダー | 会社カレンダー（土日・祝日・会社休日）と個人スケジュール（休暇・会議等）を統合した担当者別の稼働可能日・時間 |
| 予定工数 (yoteiKosu) | タスクの当初計画工数（時間単位） |
| 全体起点日 | モードに応じたデフォルトの起点日。初期計画ではプロジェクト開始日、リスケでは今日 |
| 担当者起点日 | 担当者ごとに任意で設定できる起点日。未設定の場合は全体起点日が適用される |

---

## 3. 入力

### 3.1 共通入力

| 項目 | 取得元 | 説明 |
| --- | --- | --- |
| WBS ID | ユーザー操作 | スケジューリング対象のWBS |
| モード | ユーザー操作 | `initial`（初期計画）または `reschedule`（リスケ） |
| タスクリスト | `TaskRepository.findByWbsId()` | WBSに紐付く全タスク |
| 担当者情報 | `WbsAssigneeRepository` | タスクに割り当てられた担当者 |
| 会社カレンダー | `SystemSettings.standardWorkingHours` | 標準稼働時間（CompanyCalendar） |
| 個人スケジュール | `UserScheduleRepository` | 担当者ごとの休暇・予定 |

### 3.2 モード別入力

| 項目 | 初期計画モード | リスケモード |
| --- | --- | --- |
| 全体起点日 | プロジェクト開始日（自動） | 今日（デフォルト） |
| 担当者起点日 | 任意（参画日など） | 任意 |

---

## 4. スケジューリングモード

### 4.1 初期計画モード

プロジェクト立ち上げ時に、全タスクを対象に日程を計算する。

- 全体起点日: プロジェクト開始日
- 全タスクを対象とする（タスク状態は考慮しない）
- 担当者ごとに参画日が異なる場合、担当者起点日を設定できる

### 4.2 リスケモード

プロジェクト進行中に、未完了タスクの日程を引き直す。

- 全体起点日: 今日（デフォルト）
- タスク状態に応じて処理を分岐する

| タスク状態 | 扱い |
| --- | --- |
| 完了 | スキップ。実績の開始日・終了日をそのまま使用。担当者の最終終了日を実績終了日で更新 |
| 進行中 | 開始日は実績を維持。残工数（予定工数 - 実績工数）で終了日のみ再計算 |
| 未着手 | 起点日から前詰めで開始日・終了日を計算 |

## 5. 処理フロー

### 5.1 全体フロー

```
1. WBS → プロジェクトを取得
2. WBSに紐付くタスクをタスクNo順にソート
3. モードと起点日を決定
4. 担当者ごとに前詰めでスケジュールを計算
5. 結果を一覧で返却
```

### 5.2 タスクソート

タスクはタスクNo (`taskNo`) の文字列昇順でソートされる。このソート順がスケジューリングの優先順を決定する。

### 5.3 担当者起点日の決定

```
担当者の起点日 = 担当者起点日（設定されている場合）
                ?? 全体起点日
```

### 5.4 担当者別前詰めロジック

```
担当者ごとの最終終了日 Map<assigneeId, Date> を初期化

FOR EACH task IN sortedTasks:
  IF 担当者未設定 → エラー結果を出力、SKIP

  IF mode === 'reschedule':
    IF 完了 → 結果にそのまま出力、担当者の最終終了日を実績終了日で更新、SKIP
    IF 進行中 → 残工数で終了日のみ再計算、開始日は実績を維持、SKIP

  IF 予定工数なし or 0以下 → エラー結果を出力、SKIP

  担当者の稼働カレンダーを生成（CompanyCalendar + 個人スケジュール）

  IF 担当者に前タスクの終了日がある:
    開始日 = max(前タスク終了日の翌営業日, 担当者の起点日)
  ELSE:
    開始日 = 担当者の起点日

  終了日 = 開始日から予定工数を消化できる日を計算
  担当者の最終終了日を更新
  結果を出力
```

### 5.5 終了日の計算

開始日から1日ずつ進めながら、稼働カレンダー上の稼働可能時間を予定工数から差し引く。残り工数が0以下になった日が終了日となる。

```
remainingHours = 予定工数（リスケ進行中の場合は残工数）
currentDate = 開始日

WHILE remainingHours > 0:
  availableHours = 稼働カレンダー.getAvailableHours(currentDate)
  IF availableHours > 0:
    remainingHours -= availableHours
    IF remainingHours > 0:
      currentDate を翌日に進める
  ELSE:
    currentDate を翌日に進める（非稼働日をスキップ）

RETURN currentDate
```

### 5.6 翌営業日の取得

指定日の翌日から、稼働可能時間が0より大きい最初の日を返す。

---

## 6. エラーケース

| 条件 | エラーメッセージ |
| --- | --- |
| 担当者が未設定 | `担当者が設定されていません` |
| 予定工数が未設定または0以下 | `予定工数が設定されていません` |
| 担当者情報が取得できない | `担当者が見つかりません` |
| WBSが存在しない | `WBSが見つかりません`（例外スロー） |
| プロジェクトが存在しない | `プロジェクトが見つかりません`（例外スロー） |
| プロジェクト開始日が未設定 | `プロジェクトの基準開始日が設定されていません`（例外スロー） |

> タスク単位のエラーは結果に含めて処理を継続する。WBS/プロジェクト単位のエラーは例外をスローし処理を中断する。
